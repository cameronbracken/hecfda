# hecfda 0.1.0 release design

Date: 2026-07-12
Status: approved

## Goal

Bring hecfda to a polished standalone 0.1.0 release using corehydro's 0.1.0 as the
template. The port itself is complete (all 6 phases); this release adds what corehydro
added in its own polish pass: a public API, a documentation site with worked examples,
a pixi development environment, and release hygiene (README, LICENSE, CHANGELOG,
versions, tag).

hecfda will eventually merge into corehydro. That merge is out of scope here, and
duplicating functionality corehydro already has (for example a distribution API) is
acceptable; reconciliation happens later. corehydro 0.1.0 serves purely as the design
template.

## Decisions made during brainstorming

- **API scope:** a curated workflow API around the headline pipeline, not a full
  object-model exposure and not a mere polish of the existing test shims.
- **Examples:** upstream-derived fixture scenarios retold as narrative examples, so
  every example reproduces a number validated against the real HEC-FDA C#. No
  synthetic study.
- **Template fidelity:** mirror corehydro's structure nearly verbatim (site layout,
  pixi approach, Makefile targets, changelog format) to make the eventual merge
  mechanical.

## 1. Public API

The same surface in both packages, following corehydro's convention: plain snake_case
functions that take data frames/arrays and return tidy lists/data frames. Distributions
enter the workflow functions as specifications (family name plus parameters), not as
objects.

Workflow functions, in pipeline order:

1. `stage_damage(structures, occupancy_types, hydraulics, stages, seed, ...)` --
   structure inventory table, occupancy-type definitions, and per-frequency hydraulic
   stage profiles in; stage-damage curves with uncertainty out, one per damage
   category. Wraps `ScenarioStageDamage`/`ImpactAreaStageDamage`.
2. `ead_simulation(frequency_curve, stage_damage, seed, convergence, levee = NULL, ...)`
   -- the seeded Monte Carlo EAD compute. Returns mean EAD, the EAD distribution
   (quantiles/histogram), AEP, and assurance metrics. Wraps `SimulationBuilder` +
   `ImpactAreaScenarioSimulation`.
3. `scenario_results(simulations)` -- impact-area fan-out and aggregation. Wraps
   `Scenario`.
4. `alternative_ead(base, future, base_year, future_year, period, discount_rate)` --
   EqAD annualization. Wraps `Alternative::compute_eqad`/`annualization_compute`.
5. `alternative_comparison(without, with)` -- with/without-project benefits. Wraps
   `AlternativeComparisonReport`.

Supporting exports:

- `rng_sequence()` -- the .NET-parity seeded generator, the reproducibility showpiece.
- `dist_pdf`/`dist_cdf`/`dist_quantile`/`dist_sample` -- evaluators over the 13 ported
  distribution families, addressed by specification.

The existing `hecfda_*` shims stay unexported and unchanged; the fixture runners keep
using them. The new API is a layer beside them, not a rewrite. R functions get roxygen2
docs (feeding pkgdown); Python functions get numpydoc docstrings (feeding quartodoc).

## 2. Documentation site

A Quarto website in `site/`, copied structurally from corehydro:

- Dual theme (`flatly` light, `darkly` dark) with corehydro's scss offsets,
  `freeze: auto`, code-copy. Deployed to `https://cameronbracken.github.io/hecfda/`.
- `docs.yml` GitHub Actions workflow: materialize core, install hecfdapy + doc deps,
  set up R + pkgdown, quartodoc build, quarto render, pkgdown build copied into
  `_site/r/`, deploy to Pages. Trigger paths mirror corehydro's.
- Navbar: Examples | Porting Status | Python Reference | R Reference | GitHub icon.
- Python API reference via quartodoc, sectioned: Workflow (the 5 pipeline functions),
  Distributions (`dist_*`), Reproducibility (`rng_sequence`).
- R API reference via pkgdown mounted at `/r/`.
- Landing page (`index.qmd`) kept in sync with the README.

### Porting status page

`status.qmd` in corehydro's exact format: per-area tables with
`Area | C++ core | Packages | Notes` columns and a Public/Internal distinction matching
the curated API. One table for `HEC.FDA.Statistics` (distributions, special functions,
validation, sample statistics, histograms, convergence) and one for `HEC.FDA.Model`
(paired data, structures, stage-damage, compute, metrics, scenarios, alternatives),
plus a section listing the deliberately severed surfaces (RAS-Mapper/GIS, hydraulics
grid ingest, SQLite/DBF persistence, serialization, LifeLoss, MVVM/messaging). Status
as of v0.1.0. Content condenses the existing CLAUDE.md/PLAN.md status into a
public-facing page.

### Worked examples

Each example exists in both languages (Jupyter notebook for Python, `.qmd` for R),
numbered like corehydro's, under `site/examples/`. Each states the upstream C# value it
reproduces. Plots are clean, earth-toned, and colorblind-safe.

1. **Getting started** -- install, seeded RNG parity across R/Python/C#, a first
   distribution evaluation.
2. **Distributions** -- the 13 families: evaluate, sample, fit; user-relevant
   faithful-bug caveats.
3. **Stage-damage** -- the tractable Residential/Commercial scenario from upstream
   `TractableStageDamageTests`: build the inventory and occupancy types, run
   `stage_damage()`, plot the uncertainty curves.
4. **EAD simulation** -- the deterministic oracles (150000, 83333.33, ...) then the
   seeded benchmark (`121194.5159789352` at exactly 100 iterations), showing
   convergence, AEP, and assurance outputs.
5. **Alternatives and benefits** -- the capstone: two scenarios, annualization via
   `alternative_ead`, then with/without-project benefits, reproducing upstream
   `AlternativeResults_Test` (208213.80 / 239260.18).

## 3. Environment, hygiene, and release mechanics

**Pixi.** `pixi.toml` copied from corehydro's approach: conda-forge;
`osx-arm64`/`osx-64`/`linux-64`; Python + pytest/numpy/pandas/matplotlib/jupyter +
quartodoc with the griffe pin + python-build + cmake/make/quarto/pandoc. R and dotnet
stay system-managed, with the same explanatory comment block. Tasks map 1:1 to Makefile
targets. `pixi.lock` committed and marked generated in `.gitattributes`.

**Makefile.** Add corehydro's missing targets: `build-r`, `build-py`, `docs`
(quartodoc, quarto render, pkgdown, assemble `_site/r/`), `docs-serve`.

**README.** Full rewrite replacing the stale Phase 0 content, structured like
corehydro's: pitch paragraph (unofficial port, upstream links), what's ported with a
link to the status page, R/Python packages note, Documentation links, Development
status, Install (pak / pip from git), Quick start showing the same computation in both
languages with matching seeded output, Reproducibility, Why?, AI Use and Credit,
License.

**Hygiene.** Root `LICENSE` (0BSD); `hecfdapy/LICENSE` added; `hecfdar/DESCRIPTION`
license field changed from `file LICENSE` to a proper 0BSD declaration. Root
`CHANGELOG.md` in Keep a Changelog format, three components versioned together, with a
0.1.0 entry summarizing the whole port. Versions bumped to 0.1.0 everywhere (R package,
Python package, pixi).

**Release mechanics.** Merge `phase6` into `main` first (complete and green). All
release work happens on a `release-0.1.0` branch off `main`, merged by PR. Then tag
`v0.1.0`, create a GitHub release from the changelog entry, and enable Pages on the
repo.

## 4. Testing

The new API functions get real test suites (testthat for R, pytest for Python)
asserting the same oracle numbers the examples reproduce. The existing fixture runners
stay untouched. The full exit gate must stay green: `make test-core`, `make test-r`,
`make test-py`, `make oracles`. The site build (`make docs`) is verified locally and by
the `docs.yml` workflow before merge.

## Out of scope

- The corehydro merge and any API reconciliation with it.
- CRAN/PyPI submission (the release is a git tag + Pages site).
- New numerical functionality; the port is complete and frozen for this release.
- Windows in pixi (matches corehydro; CI still covers Windows builds).
